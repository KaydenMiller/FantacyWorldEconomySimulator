# Autonomous Build — Decisions Log

This log records educated guesses and design-gap resolutions made while building the World Economy Simulation autonomously (per the user's directive to build all phases to a functioning simulation, fixing/guessing as needed and noting choices). Each entry: what was ambiguous, the choice, and why.

## Operating mode
- Build each phase → test (warning-clean build + green tests) → commit → merge to `master` → next phase.
- A **CLI** (`WorldEcon.Cli`) is built for exercising the simulation headlessly (no UI), per the user's instruction.
- Reviews are lighter in autonomous mode (self-verify + tests; deeper review only at risky integration points) to maintain velocity.

## Decisions
- **2026-06-22 — Shop Economy Phase 2: demand side & retail money (built; brainstorm→spec→plan→subagent-driven).** Added the walleted demand side (the mirror of merchants). **`RepresentativeConsumer`** (Seat/Size/Budget; Size = people represented) spawned per population (`ConsumerSpawnPhase`, weekly, `max(1, pop/1000)` per settlement). **Income behind a swappable seam:** `IConsumerIncome` (default `AllowanceIncome` = `Size × perCapitaAllowance`, a faucet) applied weekly by `ConsumerIncomePhase` — a future **wage/labor phase** swaps the strategy without touching consumer/needs/buying. **Tiered needs:** `NeedTier { Essential, Standard, Comfort }` on `Good`; consumers buy in tier-priority order. **`ConsumerDemandPhase`** (daily, Order 20) **replaces the free `ConsumptionPhase`**: each consumer buys its tiered needs from shops **cheapest-retail-price first within budget** (`stock.Withdraw` + `consumer.Spend` + `shop.CreditTill`); per-shop sales emit ONE `Trade` event per shop·good·day (Shop scope — the long-wanted "sold X to townsfolk for Y"), unmet demand a `Stockout` (Settlement scope); the `Consumed` event is **retired** (kept in the enum for parse-compat). **Scarcity-flexing retail price** (`RetailPricing`): `cost × (1 + markup × scarcityMult)` where scarcityMult uses the same supply/demand knobs as wholesale pricing — glut floors the markup toward cost, scarcity raises it (begins biting the glut problem before the Phase 4 throttle). Wholesale `MarketPrice` (set by `PricingPhase`) is unchanged = the merchant/board signal. **Key design choices:** demand runs before pricing (independent scarcity computations — retail uses Σ consumer demand, wholesale uses population+industrial); price fixed for the whole day before any buying (determinism + fairness); a consumer skips an unaffordable good and only stops the day's basket when budget is truly zero (the within-tier broke-stop fix — caught in the engine review); per-shop cost basis means retail prices differ across shops. UI: `:consumers` resource + city category + CLI `consumers`; marketplace board `Price` column now shows **retail** price, wholesale moved to the shop drill-down. Demo pre-seeds consumers (1 week's budget) + tiered goods (Cloth/Standard, Ale/Comfort) so day-1 retail works. Spec: `docs/superpowers/specs/2026-06-22-shop-economy-phase2-demand-design.md`; plan: `docs/superpowers/plans/2026-06-22-shop-economy-phase2-demand.md`. **Roadmap (next):** Phase 3 — retail restocking (wholesale→retail money flow) + industrial/trade over shops + transient caravan stalls; Phase 4 — production profitability throttle; Later — wage/labor income loop (swaps the income seam, closes the money loop), happiness→luxury demand.
- **2026-06-22 — Shop Economy Phase 1: shop substrate (built; brainstorm→spec→plan→subagent-driven).** Re-based the economy onto shops, the first phase of a multi-phase re-architecture. **Retired the `StockpileOwnerKind.SettlementMarket` pool** — all economy inventory is now shop-owned, each shop holding goods at its **own** weighted-average cost basis (fixing the unrealism that one shared market pool forced every seller to share one cost basis/price). Every `ProductionNode` and `ResourceEndowment` lazily gets a **Producer** shop (its storefront; output deposits there at production cost); each settlement gets a **PublicMarket** shop (catch-all for imports + seeded stock); existing retail shops are reclassified `Retail`. "The market" is now an **aggregate read-model** (`ShopMarket` engine helper) over a settlement's shops — supply + a uniform town reference price (Phase 1; per-shop retail pricing is Phase 2). Re-pointed Production/extraction, Pricing, Consumption, industrial input demand, Trade, Perishability, and party `adjust` onto shops via `ShopMarket` (stable shop-id ordering preserves determinism/granularity). EF migration **re-owns** legacy market stock into PublicMarket shops conserving qty/cost/price (forward-only; verified by single- and 2×2 conservation tests). TUI gains a **marketplace board** (`Good | Category | Shop | Qty | Min Price | Price` — Min Price = per-shop cost basis/break-even; one row per shop offer) + column sort. **Key choices:** generalized `Shop` (`Kind` Producer/Retail/PublicMarket) with a forward `ProducerShopId` on node/endowment (write-once); producer/public-market shops created **lazily** (mirrors the old get-or-create-stockpile pattern) so the migration only re-owns; **no retail money** (Till/MarkupBp dormant) — the Phase 1↔2 seam. Comprehensive opus-level engine review confirmed determinism, value conservation, and the trade `destPrice` fix (pay merchant at the destination's existing aggregate price, computed before depositing into a fresh zero-price stockpile). Spec: `docs/superpowers/specs/2026-06-22-shop-economy-phase1-substrate-design.md`; plan: `docs/superpowers/plans/2026-06-22-shop-economy-phase1-substrate.md`. **Roadmap (next):** Phase 2 — demand side & retail money (`RepresentativeConsumer` cohort agents with budget + needs buy from shops; tills cycle; per-shop retail pricing; OPEN: consumer income = wages vs allowance); Phase 3 — industrial/trade over shops + transient caravan stalls (farmers-market); Phase 4 — production profitability throttle (closes the glut loop). Later: happiness→luxury demand.
- **2026-06-22 — Pre-retail cleanup cluster + world-configurable currency.** A batch of polish/bugfixes plus one new persisted subsystem, ahead of the shop retail-sales feature:
  - **World-configurable currency (display only).** `CurrencyDefinition` (ordered denominations; default copper/silver/gold/platinum at 10:10:10, symbols c/s/g/p) is persisted on `World.Currency` exactly like the calendar (JSON converter + migration that backfills existing rows with the default). `CurrencyDefinition.Format(Money)` renders base-copper amounts as e.g. `3g 2s 1c` (321), `3g` (300), `50p`, `0c`, `-3g 2s 1c`. `Money` math is untouched (denominations are presentation only → zero determinism impact). Wired into all TUI money columns/details (`TuiContext.FormatMoney`) and CLI commands; quantities/counts/percentages/ticks stay raw. Plan: `docs/superpowers/plans/2026-06-22-world-configurable-currency.md`.
  - **Advance-time unit abstractions.** `CalendarSystem.TryParseDurationToTicks` accepts `m/h/d/w/M/y` (case-sensitive: `m`=minute, `M`=month), derived from the world's `CalendarDefinition` (default 1d=1440, 1w=10080, 1M=43200, 1y=518400); bare integers still = ticks. Wired into CLI `advance` and the TUI advance prompt.
  - **Work-order double-complete bug FIXED (engine determinism).** `ProductionPhase.LoadIncompleteWorkOrders` returned EF identity-resolved instances without a final `!Completed` filter, so an order completed on an earlier day of a multi-day advance (unsaved → DB still matches `WHERE Completed=0`, but EF returns the tracked Completed=true instance) re-entered `dueOrders` and `MarkComplete()` threw "already complete". Manifested as `advance 1d` then `advance 1w` throwing while a single `advance 1w` worked — i.e. a granularity-independence violation. Fixed by filtering `!Completed` on the merged result (mirrors `TradePhase.LoadUndeliveredCaravans`). Regression test added.
  - **Logging readability.** Shop-owned spoilage now logs at the shop scope (previously only market spoilage logged, so shop goods e.g. bread appeared to vanish silently — it was perishability, not a sale). `TradePhase` caravan messages now resolve good/settlement **names** instead of raw GUIDs.
  - **TUI polish.** `/` now regex-filters **any** list view (k9s-style, per-frame client-side filter over the displayed rows, ReDoS-guarded; selection/drill respect the filter), not just logs; `l log` hint is context-sensitive (only on rows with a log scope); `?` help is now a left-aligned `NavView` table pushed on the stack instead of a centered MessageBox.
  - **Still NOT built → needs design:** shop **retail-sales + till** model. The build spec (§6.3) designs shops to sell at cost×markup with a finite till that can run dry, and the event-log spec anticipated shop-sale `Trade` events, but no executed plan implemented it — consumption draws from settlement markets and `Shop.Till` is dead state. Next up via brainstorm→spec→plan (key open question: how shop retail reconciles with the existing market-based population demand; "buyer" = abstract local population).
- **2026-06-22 — Scoped activity/event log subsystem (built; brainstorm→spec→plan→subagent-driven build).** Added a unified append-only `LogEvent` stream with **write-time visibility materialization**: a `LogEventScope` row per level an event is visible at (origin + ancestors that clear the level's magnitude floor). Magnitude tiers (Routine/Notable/Major/Historic) drive both upward propagation (per-level floors + per-type force/cap overrides in `LogMagnitudePolicy`) and **granularity-independent retention** (age-relative-to-now, monotonic deletion → `advance(N)` survivors == `advance(k×N/k)`; player-action + Major/Historic never pruned). Emitted by engine phases (merchant gained/arrived/departed, production changed, stockout incl. total stockouts, market spoilage) and by player/DM actions. **`DmActionService`/`DmAction` were folded into `LogEventService`/`LogEvent`** (player-action events; old rows copied by the `AddLogEvents` migration — verified by a real-row copy test). Reads: `LogQueryService` (newest-first via denormalized `long Sequence`, in-memory regex with a ReDoS `matchTimeout`) + on-demand `SummaryService` (`ScopeSummary`). Surfaces: CLI `log`/`summary`; TUI scoped log view (`l`), `/` regex filter, `:log` root, `:summary`. **Key design choices:** raw `Guid` for `OriginId`/`ScopeId` (polymorphic, SQL-translatable) and denormalized `long Sequence` on scope rows (so the hot read `WHERE ScopeKind,ScopeId ORDER BY Sequence` runs in SQLite, sidestepping the value-converted-typed-ID translation limit); `LogEventScope` needs an **explicit `b.Property(x => x.WorldId)`** (no navigation → EF won't infer it) and a cascade-delete FK as a safety net (LogRetention is the primary prune path); Application now references Persistence/`WorldDbContext` directly (consistent with EF-as-live-state). **Invariant:** logs are write-only observability — no phase reads the log, so pruning/logging can't affect sim determinism. **Deferred (not half-built):** claim-based extra visibility, shop (non-market) spoilage, advance-time auto-summary, CLI merchant/shop scope addressing (TUI `l` reaches those). Spec: `docs/superpowers/specs/2026-06-21-activity-event-log-design.md`; plan: `docs/superpowers/plans/2026-06-21-activity-event-log.md`.
- **2026-06-21 — Confirm dialogs use the in-shell y/N prompt, not a MessageBox button dialog.** User reported the Yes|No disable-production confirm couldn't switch the active button with hjkl. In this Terminal.Gui v2 build, built-in `MessageBox` button dialogs don't honor hjkl, and the app-wide key-translation route is a dead end: `Application.KeyDownNotHandled` isn't exposed on the instance `IApplication`, and the static `Application.RaiseKeyDownEvent` is `[Obsolete]` ("the legacy static Application object is going away") — which fails the warnings-as-errors build. Rather than fight the framework, `ConfirmAsync` now routes through the same in-shell prompt bar as text/number input (`{message} (y/N)`): type `y`/`yes` to confirm; anything else, empty/Enter, or Esc-cancel means No. This is keyboard-uniform with the rest of the input model and needs no button navigation. Table hjkl (forwarded to arrow keys via `NewKeyDownEvent`) is retained. Verified end-to-end under tmux: `x`→`y`→Enter disables production (OK box), `x`→`n`→Enter cancels cleanly.
- **2026-06-22 — TUI is a k9s drill-down navigator (replaces the flat resource switcher).** Per user: Enter drills *into* the selected row's children in the **main view** (nav stack + breadcrumb), Esc/Backspace backs out, ':' jumps to any root resource. Drill chain follows containment: Continents→Countries→Regions→Cities→(category chooser: Merchants/Shops/Factories/Market)→shop goods / merchant caravans / factory outputs / market goods. ':' roots are global flat lists (regardless of parent). Built as a UI-agnostic `Navigator`/`NavView` (tested) + a Terminal.Gui shell. **Autocomplete:** a popup-below-cursor doesn't fit a bottom command bar, so used **inline ghost** (`AppendAutocomplete`) — typing "merc" shows "hants" dimmed — plus unique-prefix Enter. Verified end-to-end under tmux (`send-keys`/`capture-pane`).
- **2026-06-22 — Geography v2: nested/overlapping, country-optional, contested.** Real-campaign geography (oceans spanning continents with no country; disputed regions/cities; ruined settlements; non-land region kinds; regions that nest/overlap like a park spanning states) doesn't fit a strict tree. Evolved the model **additively** (existing factories/tests unchanged): `Region.CountryId` is now **optional** (primary/administrative country; null for oceans/wilderness) + `RegionKind` (Land/Ocean/Mountain/…); `Settlement.State` (Active/Ruined/Abandoned); `RegionContinent` (m2m region↔continent); `RegionContainment` (m2m region nesting/overlap, parent↔child); `TerritorialClaim` (country → settlement/region, Controls/Disputes). "Country → Regions" drilling = primary-country regions ∪ claimed regions. The **claims/control** piece is the lightweight precursor to the OPEN Factions system (§10) — full influence/control deferred to that design. TUI drill-nav (next) renders this model.
- **2026-06-21 — TUI input model: in-shell prompt bar, not modal dialogs (from live tmux debugging).** In Terminal.Gui 2.4.7, a modal `Application.Run(dialog)` launched from a background-dispatched `Invoke` context does NOT receive keyboard input (focus stuck on a button; typed text never lands) — but the built-in `MessageBox.Query` and in-shell `TextField`s (command bar) DO. So: text/number prompts use an **in-shell prompt bar** that resolves a `TaskCompletionSource<string?>` (Enter=value, Esc=null); messages/confirms use `MessageBox`. Threading model: command keys dispatch work to a **background thread** (EF async can't deadlock the UI sync-context), and dialogs + view updates marshal back via `app.Invoke`; the key handler returns immediately so the UI loop stays free to service them. TCS uses `RunContinuationsAsynchronously` so the action continuation doesn't resume on the UI thread. Also fixed: details `d`→Enter dismiss (was running the dialog on a background thread via `Task.Run`); added **vim hjkl** navigation (forwarded to arrow keys via `NewKeyDownEvent`). Verified end-to-end under tmux (`send-keys`/`capture-pane`): hjkl, details+Enter, advance(number), buy-out(text+number), quit.
- **2026-06-21 — k9s-style TUI on Terminal.Gui v2 (beta).** Researched: Terminal.Gui v2 is the only mature .NET full-screen TUI framework (TableView/command-bar/keybindings); Spectre.Console can't drive live interactive controls. v2 has since shipped **stable** — the 2.0.0-beta/develop builds were yanked from NuGet, so we pin **Terminal.Gui 2.4.7** (latest stable v2; namespaces split into `Terminal.Gui.App/Views/ViewBase/...`, instance `Application.Create()` API, `TableView` via `DataTableSource`). Referenced only from `WorldEcon.Tui`. Keep `WorldEcon.Cli` for scripting/automation; TUI (`WorldEcon.Tui`) becomes the primary interactive interface. **Architecture for extensibility:** a UI-agnostic core (`IResource` data providers, `IAction` row/global actions, an `IUserInteraction` prompt abstraction) separate from the Terminal.Gui shell — so (a) the core is unit-testable without a terminal, and (b) data-entry/create-edit forms are just new `IAction`s added later (v1 is query + sim-control + party actions; data entry deferred but NOT locked out, per user).
- **2026-06-21 — Determinism is snapshot/branch-based, NOT seed-re-import-based (known scope; from final review).** Entity IDs are random `Guid.NewGuid()`, and engine phase ordering + pathfinder/trade tie-breaks key on `Id.Value`. Consequence: importing the SAME seed file twice yields divergent worlds (different GUIDs → different tie-break order when nodes contend for a scarce input, etc.). What DOES hold and covers the spec's actual determinism use-cases: **granularity-independence** (advance 2880 == 2×1440 on the same DB) and **branch/compare/what-if** (file-copy snapshots preserve GUIDs, so a branched world replays identically). Full seed-reproducibility ("same seed ⇒ same world from scratch", spec §3.3) requires **seeded ID generation** — derive all entity IDs from a dedicated seeded RNG stream during import AND sim-entity creation. Deferred as a substantial refactor; until then, reproducibility = snapshot-based. (Verified by final review: core is otherwise sound — phase composition coherent, no double-count, `.Local`-merge correct everywhere, conservation invariants hold.)
- **2026-06-21 — Junction nodes deferred past trade (deviation from Build §12.1).** §12.1 recommended landing the settlements∪junctions node type *before* trade/pathfinding. To avoid a Route-endpoint refactor mid-build, I build pathfinding + trade over the **settlement-only** graph first (Routes already connect SettlementId↔SettlementId). Junction nodes remain a planned fidelity enhancement (shared-leg danger/traffic) to add later; pathfinding is written so a node-type generalization is possible.
- **2026-06-21 — Trade is deterministic first; RNG-based events deferred.** First trade implementation moves goods down price gradients deterministically (no dice), so RNG-state persistence across advances isn't needed yet. Caravan ambush/failure and other RNG events come later, at which point `IRngStreams.Capture()` gets persisted on `World` and restored in `SimulationContext.LoadAsync`.
- **2026-06-21 — Engine phases must merge EF `Local` with DB queries (important pattern).** The engine saves once at the end of a multi-tick advance, so entities created earlier in the same advance (e.g. a WorkOrder started on day N) are EF-`Added` but unsaved. Phase queries MUST union `ctx.Db.<Set>.Local` with the DB query (dedup by Id) — otherwise a multi-day advance behaves differently from stepping day-by-day. Also: value-converted types (`Tick`, typed IDs) don't translate in SQL `WHERE`/`ORDER BY`, so filter/sort those client-side after materializing. Verified `advance 2880` == 2× `advance 1440` for production. All future phases follow this.
- **2026-06-21 — Tick engine uses EF-tracked entities as live state (major).** Spec Build §4 envisioned an in-memory POCO engine + SQLite *projection* + action-log-as-source-of-truth with replay. For a single-user desktop sim, I instead make the **SQLite DB the live simulation state**: the engine loads entities via `WorldDbContext`, mutates them, advances `World.CurrentTick`, and `SaveChanges`. Determinism comes from the seeded RNG + stable iteration order (explicit `OrderBy`); branch/compare/what-if already work via the file-copy snapshot/branch/compare built in Plan 1b (which need no log). **Trade-off:** we give up O(1) log-fork branching and replay-under-different-ruleset; those can be layered later by adding log-replay on top. The `DmAction` log (Plan 4) is still built as the party-effect channel + audit record, but is not (yet) the source of truth. Rationale: dramatically simpler, gets to a functioning sim, determinism still holds for forward runs.
- **2026-06-21 — Tick engine lives in a new `WorldEcon.Engine` project** (refs Domain + Persistence + Simulation). `WorldEcon.Simulation` stays pure-math (calendar/RNG/fixed-point); orchestration that needs persistence lives in Engine. Phases mutate tracked entities via a `SimulationContext`; tested against SQLite like the rest of the suite.
- **2026-06-21 — RNG-state persistence deferred until a subsystem draws from RNG.** Production/consumption/pricing are deterministic functions of state, so they need no RNG. `IRngStreams` is seeded from `World.Seed` and wired into the context now; persisting/restoring stream positions across advances is required before **trade/events** (which roll dice) and will be added then (store `RngState` per stream on `World` or a side table).
- **2026-06-21 — Cost-basis overflow policy.** Resolved the deferred `Money`/`FixedMath` overflow-policy question by making cost-basis accumulation **loud** (`checked`) — consistent with `FixedMath`. Silent wrong cost basis would corrupt the deterministic economy. (Plan 2 T3.)
