# Autonomous Build — Decisions Log

This log records educated guesses and design-gap resolutions made while building the World Economy Simulation autonomously (per the user's directive to build all phases to a functioning simulation, fixing/guessing as needed and noting choices). Each entry: what was ambiguous, the choice, and why.

## Operating mode
- Build each phase → test (warning-clean build + green tests) → commit → merge to `master` → next phase.
- A **CLI** (`WorldEcon.Cli`) is built for exercising the simulation headlessly (no UI), per the user's instruction.
- Reviews are lighter in autonomous mode (self-verify + tests; deeper review only at risky integration points) to maintain velocity.

## Decisions
- **2026-06-21 — Engine phases must merge EF `Local` with DB queries (important pattern).** The engine saves once at the end of a multi-tick advance, so entities created earlier in the same advance (e.g. a WorkOrder started on day N) are EF-`Added` but unsaved. Phase queries MUST union `ctx.Db.<Set>.Local` with the DB query (dedup by Id) — otherwise a multi-day advance behaves differently from stepping day-by-day. Also: value-converted types (`Tick`, typed IDs) don't translate in SQL `WHERE`/`ORDER BY`, so filter/sort those client-side after materializing. Verified `advance 2880` == 2× `advance 1440` for production. All future phases follow this.
- **2026-06-21 — Tick engine uses EF-tracked entities as live state (major).** Spec Build §4 envisioned an in-memory POCO engine + SQLite *projection* + action-log-as-source-of-truth with replay. For a single-user desktop sim, I instead make the **SQLite DB the live simulation state**: the engine loads entities via `WorldDbContext`, mutates them, advances `World.CurrentTick`, and `SaveChanges`. Determinism comes from the seeded RNG + stable iteration order (explicit `OrderBy`); branch/compare/what-if already work via the file-copy snapshot/branch/compare built in Plan 1b (which need no log). **Trade-off:** we give up O(1) log-fork branching and replay-under-different-ruleset; those can be layered later by adding log-replay on top. The `DmAction` log (Plan 4) is still built as the party-effect channel + audit record, but is not (yet) the source of truth. Rationale: dramatically simpler, gets to a functioning sim, determinism still holds for forward runs.
- **2026-06-21 — Tick engine lives in a new `WorldEcon.Engine` project** (refs Domain + Persistence + Simulation). `WorldEcon.Simulation` stays pure-math (calendar/RNG/fixed-point); orchestration that needs persistence lives in Engine. Phases mutate tracked entities via a `SimulationContext`; tested against SQLite like the rest of the suite.
- **2026-06-21 — RNG-state persistence deferred until a subsystem draws from RNG.** Production/consumption/pricing are deterministic functions of state, so they need no RNG. `IRngStreams` is seeded from `World.Seed` and wired into the context now; persisting/restoring stream positions across advances is required before **trade/events** (which roll dice) and will be added then (store `RngState` per stream on `World` or a side table).
- **2026-06-21 — Cost-basis overflow policy.** Resolved the deferred `Money`/`FixedMath` overflow-policy question by making cost-basis accumulation **loud** (`checked`) — consistent with `FixedMath`. Silent wrong cost basis would corrupt the deterministic economy. (Plan 2 T3.)
